# Dashboard UI Foundation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the current placeholder dashboard with a merchant-first shadcn-based shell foundation, custom visual system, providers, navigation metadata, and implementation documentation.

**Architecture:** Keep the Next.js App Router app in `apps/dashboard`, isolate interactive providers in client components, and build app chrome from shadcn primitives. Feature pages remain empty operational routes; real product/order/auth/onboarding workflows are out of scope for this pass. The custom shadcn base/preset direction is created by this project; the plan does not assume an existing preset code from the user.

**Tech Stack:** Next.js 16 App Router, React 19, shadcn/ui, Tailwind CSS, Radix-based shadcn components, `next-themes`, `@remixicon/react`, TanStack Query, TanStack Table conventions, TanStack Form conventions, Zod, TypeScript.

```ts
import { z } from "zod";

export const listSearchParamsSchema = z.object({
  page: z.coerce.number().int().min(1).catch(1),
  pageSize: z.coerce.number().int().min(10).max(100).catch(20),
  q: z.string().trim().catch(""),
  sort: z.string().trim().catch(""),
  view: z.string().trim().catch(""),
});

export type ListSearchParams = z.infer<typeof listSearchParamsSchema>;

export function parseListSearchParams(input: URLSearchParams | Record<string, string | string[] | undefined>) {
  if (input instanceof URLSearchParams) {
    return listSearchParamsSchema.parse(Object.fromEntries(input.entries()));
  }

  const normalized = Object.fromEntries(
    Object.entries(input).map(([key, value]) => [key, Array.isArray(value) ? value[0] : value]),
  );

  return listSearchParamsSchema.parse(normalized);
}
```

```ts
export type ApiErrorKind =
  | "auth"
  | "permission"
  | "validation"
  | "conflict"
  | "not-found"
  | "rate-limit"
  | "server"
  | "network";

export class DashboardApiError extends Error {
  constructor(
    message: string,
    public readonly kind: ApiErrorKind,
    public readonly status?: number,
  ) {
    super(message);
    this.name = "DashboardApiError";
  }
}

export function apiErrorKindFromStatus(status: number): ApiErrorKind {
  if (status === 401) return "auth";
  if (status === 403) return "permission";
  if (status === 404) return "not-found";
  if (status === 409) return "conflict";
  if (status === 422) return "validation";
  if (status === 429) return "rate-limit";
  return "server";
}
```

```ts
import { DashboardApiError, apiErrorKindFromStatus } from "@/lib/api/errors";

export async function dashboardFetch<T>(path: string, init?: RequestInit): Promise<T> {
  let response: Response;

  try {
    response = await fetch(path, {
      ...init,
      headers: {
        "content-type": "application/json",
        ...init?.headers,
      },
    });
  } catch (error) {
    throw new DashboardApiError(error instanceof Error ? error.message : "Network error", "network");
  }

  if (!response.ok) {
    throw new DashboardApiError(
      `Request failed with status ${response.status}`,
      apiErrorKindFromStatus(response.status),
      response.status,
    );
  }

  return response.json() as Promise<T>;
}
```

```tsx
"use client";

import { useTheme } from "next-themes";
import { useEffect, useState } from "react";

import { AppIcons } from "@/components/app/icons";
import { Button } from "@/components/ui/button";

export function ThemeToggle() {
  const { resolvedTheme, setTheme } = useTheme();
  const [mounted, setMounted] = useState(false);
  const [transitionPoint, setTransitionPoint] = useState<{ x: number; y: number } | null>(null);

  useEffect(() => setMounted(true), []);

  const isDark = mounted && resolvedTheme === "dark";
  const Icon = isDark ? AppIcons.sun : AppIcons.moon;

  function toggleTheme(event: React.MouseEvent<HTMLButtonElement>) {
    const prefersReducedMotion = window.matchMedia("(prefers-reduced-motion: reduce)").matches;

    if (prefersReducedMotion) {
      setTheme(isDark ? "light" : "dark");
      return;
    }

    setTransitionPoint({ x: event.clientX, y: event.clientY });
    window.setTimeout(() => setTheme(isDark ? "light" : "dark"), 90);
    window.setTimeout(() => setTransitionPoint(null), 320);
  }

  return (
    <>
      <Button
        type="button"
        variant="ghost"
        size="icon"
        aria-label="Toggle theme"
        onClick={toggleTheme}
      >
        <Icon />
      </Button>
      <div
        aria-hidden="true"
        className="theme-transition-overlay"
        data-active={transitionPoint ? "true" : "false"}
        style={
          transitionPoint
            ? ({
                "--theme-x": `${transitionPoint.x}px`,
                "--theme-y": `${transitionPoint.y}px`,
              } as React.CSSProperties)
            : undefined
        }
      />
    </>
  );
}
```

```md
# Dashboard UI Foundation

The dashboard is a merchant-first Next.js app. It uses shadcn/ui for shell primitives and custom tokens for a polished, blue-accented operations interface.

## Scope

The foundation pass builds the dashboard shell, visual system, providers, route structure, navigation metadata, command center, theme handling, and documentation.

It does not build product management, order management, auth, onboarding, billing workflows, analytics charts, or platform-operator tools.

## Shell

- Desktop uses a persistent shadcn sidebar.
- The sidebar can collapse to an icon rail.
- Mobile uses the shadcn sidebar mobile behavior.
- Account/profile controls live in the sidebar footer.
- The header contains sidebar trigger, breadcrumbs, command trigger, theme toggle, and future page action slots.

## Navigation

Navigation is defined in `apps/dashboard/src/lib/navigation.ts`.

The first shell renders flat navigation:

- Overview
- Products
- Orders
- Storefront Editor
- Insights
- Billing
- Settings

The metadata supports one-level child routes for later pages such as collections, categories, templates, pages, and publish history. Child submenus should only render for implemented routes.

## Visual System

- Primary accent is custom blue.
- Tokens use OKLCH.
- Neutrals are tinted, not pure black or pure white.
- Motion uses shared duration and easing tokens.
- Remix Icon is preferred for dashboard-facing icons.
- shadcn generated internals should remain compatible with the CLI.

## Theme

`next-themes` owns theme persistence and the `dark` class. The app `ThemeToggle` owns the visual transition.

The preferred transition is an ink/radial fill from the click point. Reduced motion must switch instantly.

## Data Patterns

- TanStack Query owns server state.
- Feature modules own query keys.
- Tenant-scoped queries must include tenant identity once auth and membership exist.
- Components should use API helpers instead of calling `fetch` directly.
- API errors should normalize into auth, permission, validation, conflict, not found, rate limit, server, or network categories.

## Tables

TanStack Table should power serious list pages.

Record-affecting state belongs in the URL:

- page
- page size
- search
- sort
- filters
- selected view

Presentation-only state can stay local.

## Forms

Use TanStack Form, Zod, and shadcn field components. Keep forms in feature modules.
```
